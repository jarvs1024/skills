# 下周工作计划生成规则

下周计划**默认**从当周工作内容延续推断,但**必须**在落盘前与用户确认。

## 默认推断逻辑 (LLM 自动做)

从当周工作里**带进度的子项**找延续:

| 当周状态 | 推断的下周动作 |
|---|---|
| `进度:进行中` (无完成迹象) | 继续推进, 加进度修饰 `继续 ... (上周 X%)` |
| `进度:进行中` (有部分完成) | 推进 + 收尾: `推进 ... 完成 ...` |
| `进度:20%` / `50%` 等 | 推进到下一里程碑: `推进 ... 到 50%` |
| 风险:有阻塞项 (等开发/等回复) | 跟进阻塞: `跟进 ... 回复`, 标 P0 |
| 当周完成的 entry | 升级或新方向: `基于 ... 进一步 ...` |
| 提的 issue 还没关 (#412 等) | 关闭 issue: `修复 #412, 关闭 #X` |
| 文档/准入准出 进行中 | 推进定稿: `完成 ... 定稿, 同步 ...` |

**优先级 P0 / P1 / P2 划分**:

- **P0**: 阻塞项、有 deadline (明天/周五/本周)、关键路径上的事
- **P1**: 重要但可缓冲 (本周内做完即可)
- **P2**: 锦上添花, 排不上 P0/P1 才进

## 与用户确认 (必做)

**在写入 MD 之前, 必须把推断出的下周计划展示给用户, 让用户确认或修改**。不允许 LLM 一拍脑袋直接定稿。

确认流程:

1. LLM 列出推断的 3~7 条下周计划 (带优先级)
2. 展示给用户:
   ```
   根据本周进度, 我推断了下周计划:
   1. P0  推进 BIB_V1.4.3 154 转测适配,继续 bics8 兼容性测试 (上周 20%), 解决 fw 初始化 bug
   2. P0  跟进固件 3.1 版本测试遗留的 10 个问题, 优先解阻塞项 (fw init 命令不可用)
   3. P1  修复 #413 回归脚本报错, 跑完 P99 latency 全量回归, 关闭 #412
   4. P1  推进《老化准入准出》v0.3 完成审阅, 定稿后同步产品与硬件组
   5. P2  起草新版本准入准出 v0.4 (基于本周 v0.3 反馈)

   确认这样吗? 要加 / 删 / 改哪条? (回 "OK" 直接落盘, 或 "改成: ..." 调整)
   ```
3. 用户回答:
   - `OK` / `确认` / `可以` → 直接落盘
   - `改成: ...` / `加: ...` / `删: 第 N 条` → 修改后再展示一次
   - `你来定` → LLM 按上面规则继续
   - `空着` / `这周没计划` → 下周计划表只写表头, 不写数据行
4. **不允许**在没有用户回应的情况下直接落盘下周计划

## 下周计划表的格式

| 序号 | 优先级 | 工作计划 |
|---|---|---|
| 1 | P0 | 推进 BIB_V1.4.3 154 转测适配,继续 bics8 兼容性测试 (上周 20%),解决 fw 初始化 bug |
| 2 | P0 | 跟进固件 3.1 版本测试,优先解决阻塞项 (fw init 命令不可用),关闭 10 个遗留问题 |
| 3 | P1 | 修复 #413 回归脚本报错,跑完 P99 latency 全量回归,关闭 #412 |
| 4 | P1 | 推进《老化准入准出》v0.3 完成审阅,定稿后同步产品与硬件组 |

**格式约束**:

- 序号从 1 开始, 连续整数
- 优先级只用 **P0 / P1 / P2** (不用 P3, 不用高/中/低)
- "工作计划" 列是**完整的一句话**, 不拆子条目 (与"本周总结"不同)
- 必要时**可以保留版本号 / issue 号 / 文档版本**这些 ID
- 必要时可以**保留"上周 X%"这种进度引用**, 体现延续
- 不带日期 (跟本周总结的"三列不带日期"规则一致)
- 风险点列在"下周计划"表里**不存在** (下周是计划, 不是已经发生的事)
