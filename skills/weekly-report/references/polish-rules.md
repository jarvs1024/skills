## 用户身份

**周报作者**: SSD 测试工程师 (固件 / 老化 / 性能 / 兼容性方向)
**周报读者**: 项目经理 + 上下游协作同事 (开发、硬件、产品、QA lead)

LLM 在生成周报时, 默认从"SSD 测试工程师"这个专业视角出发, 可以:

- 用领域术语 (见下方"领域词表")
- 适当润色表述, 让成果更专业、更具体
- 把口语化的工作条目**重写**为正式的工程描述

**但必须**:

- 不编造事实 (见"禁止虚构"段)
- 不改变技术含义
- 不夸大成果

## 领域词表 (可直接使用, 不需要解释)

### 测试 / 验证类

| 口语 | 润色后 |
|---|---|
| 跑测试 | 跑回归 / 执行测试用例 / 跑验证 |
| 跑通 | 完成验证 / 通过回归 |
| 跑挂 | 回归失败 / 发现 N 个异常 |
| 提了 issue | 提 issue #X (异常描述) |
| 写脚本 | 开发自动化测试脚本 / 编写测试工具 |
| 测一下 | 适配验证 / 兼容性测试 |

### 性能 / 数据类

| 口语 | 润色后 |
|---|---|
| 跑 latency | 跑 latency 回归, 关注 P50 / P99 / P99.9 |
| 数据异常 | 数据异常值, 偏离基线 N% |
| 慢 | 性能劣化 / 时延劣化 |
| 快 | 性能提升 |
| 吞吐 | IOPS / 吞吐 (MB/s) |

### 固件 / 平台类

| 口语 | 润色后 |
|---|---|

| 口语 | 润色后 |

## 润色强度 (3 档)

LLM 根据 entry 的**信息密度**自动选择:

- **轻度润色** (1 词替换): entry 本身就很专业, 只把"跑通"换成"通过回归"
  - 例: "6/16 跑通 BIB_V1.4.1" → "BIB_V1.4.1 完成回归验证"
- **中度润色** (句式重组): entry 是动作列表, 整合为正式描述
  - 例: "6/15 跟硬件对 trim 状态机,6/16 跟开发 debug 初始化" → "联合硬件/开发团队, 定位 FW trim 状态机与初始化问题"
- **重度润色** (补充上下文): entry 很口语, 需要补全技术含义
  - 例: "测了一下 154" → "BIB_V1.4.3 平台 154 颗粒兼容性测试"

## 禁止虚构 (红线)

LLM **绝对不能** 在润色时:

- 添加 entry 中**没有**的测试数据 (如"P99 提升 20%", entry 里没说)
- 虚构技术细节 (如"通过 fio 的 ioengine=psync 模式", entry 里没提)
- 编造协作对象 (如"与张工联合 debug", entry 里只有"开发")
- 修改 entry 的语义方向 (如"进行中" 不能改成"已完成")
- 改写 issue / PR 号 (用户写的什么就用什么)
- 改变版本号 (用户写的 V1.4.3 就不准改成 V1.4.2)
- 把"风险"改成"已完成"或反之

## 自我检查 (生周报后)

在落盘 MD 之前, LLM 应该自检:

1. 每个工作内容, 跟原 entry 说的是不是同一件事?
2. 关键数字 (issue 号 / 版本号 / 数据) 是不是原样保留?
3. 进度状态 (进行中/已完成/阻塞) 是不是跟原 entry 一致?
4. 风险点是不是原 entry 提到的风险, 没有添加新风险?

任何一条不过, 回到原 entry 重写。

## 跨周一致性

如果用户在工作流里**反复提到同一个工作主题** (如"BIB 154 适配"), 保持**术语一致**:

- 不要这周写"BIB_V1.4.3 154 适配",下周写"BIB 1.4.3 转测"
- 第一次出现时用全称, 之后可省略 (如"BIB_V1.4.3 154 适配" → 之后"BIB 154 适配")

如果跨周复用同一段"工作内容"描述, 用户后续修改进度即可, 不要改写内容。
