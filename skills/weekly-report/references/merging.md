# 工作模块归并规则

LLM 生周报时,默认会把每条流水账拆成一个独立模块,导致 5~10 个模块并列。本规则把模块数压到 **3~5 大类**,并把所有同类子内容合并到**一个 Excel 行**,用 `1) 2) 3)` 编号 + `<br>` 换行显示。

- ## 合并 vs 不合并的边界 (重要)

LLM 容易**过度归并**(把不相干的事合一起) 或**归并不够**(分太细)。下面给明确边界:

### 应该合并的 (Yes, merge)

- ✅ 同一**项目** / **PR** / **issue** 的多个动作
  - "提 issue #412" + "提 PR 修 #412" + "跟进 PR 合并" → `跟进 issue #412`
- 同一**文档**的多个阶段
  - "起草 v0.2" + "补全异常场景" + "推进评审" → `准入准出文档`
- 同一**协作对象**的多次沟通
  - "跟硬件对 X" + "跟硬件对 Y" → `跨方协作 (硬件)`
- 同一**测试对象** / **固件版本**的多项验证
  - "跑 P99 回归" + "跑 fio 脚本" + "升级 FW 验证" → `SSD 老化与回归`
- 同一**时间段**的连续动作
  - "上午 A, 下午 B" 同一主题 → 合并

### 不应合并的 (No, keep separate)

- ❌ 不同**协作对象**的独立工作
  - "跟硬件对 trim 状态机" (FW 主题, 归 SSD 老化) vs "跟产品对 Q3 验收" (产品/标准, 归准入准出)
- ❌ 不同**生命周期**的工作
  - "测试执行" (短周期, 几小时) vs "准入准出文档" (跨周, 长周期)
- ❌ 强**领域差异**
  - "提 issue" (开发协作) vs "review PR" (code review) — 视上下文:
    - review 别人的 PR → 单独成 `研发协作` 或 `Code Review`
    - 提 issue 测试中发现的问题 → 归测试大类
- ❌ 强**角色差异** (读者会关心)
  - "跟开发联合 debug" (技术细节) vs "跟产品对 Q3 验收" (需求细节) — 即使都是"沟通",读者关心角度不同, 拆开
- ❌ 用户**明确分开的**
  - 写周报时, 用户说 "这条和那条是分开的, 别合" → 不合

### 反向: 不要为了凑数强行合并

如果某条 entry **真的找不到同类**, 不要为了填大类强行合:

- 单条 entry 单独成 1 个大类 (1.1)
- 或者, 归到主题**最相关**的现有大类下, 哪怕只 1 条
- 但要诚实地**不编号** (按"子条目数 = 1 时不编号"规则)

## 核心原则

**大类名由 LLM 根据当周实际工作内容决定**——不要预设固定类目。每个用户/每周工作面不同,大类名应贴合当周实际出现的所有工作面。

## 归并判断流程

对每条流水账 entry,问:

1. **它本质属于哪个工作面?**(测试执行 / 文档 / 跨方协作 / 流程规范 / 工具基建 / ...)
2. **哪几条 entry 属于同一个工作面?**
3. **给这个工作面起一个 2~6 字的概括名**(后续所有同类 entry 都用这个名字)
4. 重复 1~3,直到所有 entry 都归到某个大类

## 大类命名约束

- 长度 **2~6 个汉字**
- 用**当周实际出现的工作主题**命名(如 `SSD 老化回归` / `准入准出文档` / `跨方协作`),不沿用历史周报里的大类名
- 同一大类下挂 **2 条以上** entry 才划算(单独一条不强行合并,可单列大类)
- 大类总数控制在 **3~5 个**;少于 3 个说明周报太空,多于 5 个说明归并不够

## 表格输出格式(关键)

**每个大类 = 一行 Excel**,**多个子内容用 `1) 2) 3)` 编号 + `<br>` 换行**,全部塞到"工作内容与成果"列的同一个单元格。

```markdown
| 序号 | 工作模块 | 工作内容与成果 | 完成进度 | 风险点 |
|---|---|---|---|---|
| 1 | SSD 老化与回归 | 1) 写 fio 随机写 latency 脚本,支持顺序/随机/混合三种负载<br>2) 跑 3 块样品 P99 latency 回归,发现 2 个异常值;提 issue #412(P99 异常)、#413(回归脚本报错)<br>3) 老化机 #7 升级 FW 完成后回归通过<br>4) 跟硬件组对 FW trim 状态机问题 | 进行中 | 等开发定位 #412 根因 |
| 2 | 研发协作 | review 小李 PR #218,提 4 条建议(变量命名、异常处理等) | 已完成 |  |
| 3 | 准入准出文档 | 推进《老化准入准出》v0.3 起草,补全异常场景清单 | 进行中 |  |
```

**关键约束**:

- **每个大类一个 `<tr>`**(一行 Excel),**不是**父子行折叠
- **序号列填整数 1, 2, 3**(不写 1.1 / 1.2)
- **工作内容与成果列**:用 `1) 2) 3) ...` 编号子条目,子条目之间用 `<br>` 强制换行(HTML/xlsx 渲染时显示成多行)
- **完成进度和风险点列**:填到**整行**(代表整个大类的整体进度)。如果大类下不同子条目进度差异大,只填**最差那个**(更醒目)。风险点同理。
- **行高自动撑开**:`<br>` 在 xlsx/HTML 中都会被识别为换行,`generate_xlsx.py` 的 `wrap_text=True` 已配置

## 子条目编号规则

- **子条目数 = 1 时,不加 `1)` 编号**,直接写内容
- **子条目数 ≥ 2 时**,从 `1)` 开始编号,顺序按**时间先后**排(早的在前)
- 编号前缀用全角 `)`(半角 `)` 也可,保持一致即可)
- 编号后**必须有空格**
- 子条目的内容前面**不要写日期**(见上节)

正反对照:

| 情况 | 错 | 对 |
|---|---|---|
| 1 条子内容 | `1) review 小李 PR #218,提 4 条建议` | `review 小李 PR #218,提 4 条建议` |
| 2 条子内容 | `review 小李 PR #218,提 4 条建议<br>跟进 PR 合并并验证` | `1) review 小李 PR #218,提 4 条建议<br>2) 跟进 PR 合并并验证` |
| 3 条子内容 | `1) 写 fio 脚本<br>2) 跑回归` | `1) 写 fio 脚本<br>2) 跑回归` (不变) |

## 三列都不带日期

**"工作内容与成果"、"完成进度"、"风险点" 三列都不要写 "6/15"、"6/16"、"周一" 这类具体日期或星期**。日期范围已经体现在元信息行的"本周工作总结(6.12 ~ 6.18)",三列只描述**做了什么 / 进展到哪 / 阻塞在哪**。

例外:**版本号、issue/PR 号、文档版本、固件版本号**可以保留(它们是 ID,不是日期):
- ✅ `BIB_V1.4.3 154 转测适配` (版本号)
- ✅ `issue #412、#413` (PR/issue 号)
- ✅ `v0.3 起草` (文档版本)
- ✅ `FW 3.1 版本测试` (固件版本号)
- ❌ `6/15 写脚本`
- ❌ `周一跑了回归`
- ❌ `6/17 review 阻塞` (进度列里写日期)
- ❌ `6/18 等开发回复` (风险列里写日期)

这样做的目的:

- 周报篇幅更紧凑,读者扫一眼就能看到核心产出
- 避免和"元信息行"的日期范围重复
- 让"工作内容"成为可复用的成果描述(下次复用同一段文字时不用改日期)

## 常见归并模式(参考,不是强制)

下列模式**经常**一起出现,生周报时优先考虑合并:

- **同一项目的多个时间点**: "6/15 写脚本" + "6/16 跑回归" + "6/17 升级 FW" → `SSD 老化回归`
- **同一文档的多个阶段**: "起草 v0.2" + "补全异常场景" + "推进评审" → `准入准出文档`
- **同一协作对象的多次沟通**: "跟硬件组对 X" + "跟开发联合 debug" → `跨方协作`
- **同一 PR / issue 的多个动作**: "提 issue" + "提 PR fix" + "review PR" → 看主题:
  - 主题是测试 → 归测试大类
  - 主题是代码评审 → 单独成 `研发协作` 或 `Code Review`

## md 文档结构 (必须)

LLM 写 `周报-YYYY-MM-Www.md` 时, **必须**含两个 H2 小标题, 跟 xlsx 的 sheet 标题对齐:

- `## 本周工作总结 (M/D ~ M/D)` — 包本周总结 5 列表 (序号 / 工作模块 / 工作内容与成果 / 完成进度 / 风险点)
- `## 下周工作计划 (M/D ~ M/D)` — 包下周计划 3 列表 (序号 / 优先级 / 工作计划)

H2 不可省略. xlsx 这边脚本自动写 sheet 标题, md 这边**只能靠 LLM 显式写**, 不写 = 用户看不到 "这是本周总结 / 下周计划" 段落标记.

## 渲染器支持

- `scripts/generate_xlsx.py` 已自动处理 `<br>` 换行、列宽 (按最长子行算) 和 Windows 兼容
- **不需要再改任何渲染器** — 这套格式直接走 `generate_xlsx.py`
